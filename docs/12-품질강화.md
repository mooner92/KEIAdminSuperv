# 12 · 품질 강화 (평가 · 검수 · 별표 · 리랭커) — P1.1~P1.4

> RAG 비서를 "그럴듯함"에서 **측정 가능한 정확함**으로 끌어올리는 품질 하드닝 트랙.
> WORKPLAN의 P0~P5(레포→운영 파이프라인)와 별개로, 운영 중 품질을 끌어올리는 **P1.1~P1.4** 작업이다.
> 코드: [`../eval/`](../eval/README.md) · 기준: [05-rag-design.md](05-rag-design.md) · 거버넌스: [07-security-governance.md](07-security-governance.md).

## ⛔ 이 트랙의 절대 가드레일
1. 규정 값(금액·한도·기한)·정답을 **지어내지 않는다.** 평가셋 정답·기대출처는 **실제 청크에서 추출**, 사람 확정 전 `verified:false`.
2. **검수 완료(`검수상태`) 표시는 사람만 한다.** 에이전트는 큐·도구·후보만 만들고 **자동 확정 금지**.
3. 재임베딩/재인덱싱 전 **Chroma 백업 필수**: `cp -r tools/chroma tools/chroma.bak.$(date +%F)`. 롤백 경로 기록.
4. 큰 모델(리랭커 등) 다운로드 전 **무엇/용량/온프레미스 여부 한 줄 보고** 후 진행. **외부 API 금지**.
5. 별표 등 원문 **비변형**(숫자·표 의역 금지). OCR/VLM 불확실하면 `검수상태:미검수` 유지.
6. **개선 주장은 평가셋 지표(before/after) 없이는 금지.**

## P1.1 — 평가셋 · 평가 하베스트 ✅
- **평가셋** `eval/golden.jsonl`(gitignore, 로컬 전용): 31문항 / 9개 카테고리, 전부 `verified:false`(미검수 초안).
  기대출처(`expected_sources`)는 코퍼스 실재 `(규정명, 조)`에서만 도출 — 규정 값 무추측.
- **하베스트** `eval/run_eval.py` + `eval/run.sh`: 각 질문을 실제 `rag_core.retrieve`에 통과 →
  - Hit@k / Recall@k / MRR (**strict**=규정명+조, **relaxed**=규정명만), 카테고리별 분해, 질문별 상세.
  - 선택 `--judge`: 생성 답변의 **근거 충실도**(근거 밖 사실 단정 여부)·출처표기·면책문구 LLM 판정.
  - 리포트 `eval/reports/<ts>.json`(gitignore). 평가는 **CPU**로 돌려 라이브 서빙(GPU)과 비경쟁.
- **베이스라인**(KURE-v1 + Chroma, 31문항):

  | cutoff | Hit(strict) | Recall | MRR | Hit(relaxed) |
  |---|---|---|---|---|
  | @1 | 0.677 | 0.677 | 0.677 | 0.903 |
  | @3 | 0.935 | 0.935 | 0.790 | 1.000 |
  | @5 | 1.000 | 1.000 | 0.803 | 1.000 |

  → 올바른 **규정**은 거의 top-1(0.903), 정확한 **조**는 top-3(0.935)/top-5(1.000)에 회수.
  **strict@1(0.677) ↔ relaxed@1(0.903)** 격차 = "규정 맞고 조 틀림" → **리랭커(P1.4) 개선 목표 지점**.

- **생성 품질**(LLM-judge, 적중 31 + 부정 5): 근거충실 0.935 · 출처표기 0.968 · **면책문구 0.806** · 거부율 0.800 · 근거없는 조문인용 0건.
- **관찰(사람 확인) → 조치:**
  1. ✅ **[조치완료] 면책 문구 누락 ~19%**(0.806): 14B가 마무리 "최종 판단은…"을 종종 빠뜨림 →
     `rag_core._ensure_disclaimer`로 **answer/answer_stream 모두 결정적 보장(0.806→1.000 by construction)**.
     단위테스트 `tools/test_rag_core.py`(6건) + 라이브 3100 확인. ⛔가드레일 강화(약화 아님).
  2. **환각 1건(n03)**: 길라잡이에 동호회 '명단'만 있고 금액이 없는데 "월 1만원"을 지어냄 → 거부 실패. 가장 위험한 모드를 평가셋이 포착. 근거 충실 강화(프롬프트/리랭커)로 줄일 대상.
  3. **LLM-judge 자체 오판 2건**(q004·q005): 근거에 실제 존재하는 사실을 judge가 '없음'으로 오판 → faithfulness는 **스크리닝 신호**일 뿐 확정은 사람. judge 프롬프트 개선 여지.

> 보안: `golden.jsonl`·`reports/`는 규정 원문 값을 담아 **공개 레포에 커밋하지 않는다**(`chroma`/`app.db`와 동급 로컬 데이터). 형식은 `eval/golden.example.jsonl`.

## P1.2 — 검수 큐 · 검수 도구 · 단건 재임베딩 🟡 (도구 완료, 검수는 사람 진행)
- ✅ **우선순위 큐** `tools/review_queue.py`(읽기 전용): 유형(규정30>가이드12>시스템8>용어6) + 별표/별지 +15
  + 미분류 +8 + 피인용(인바운드 [[링크]]) 가중. 미검수 274건 중 핵심 규정(인사규정 인바운드 69·복무규정 48·
  여비규정 22 등 별표 포함)이 상위로. 출력은 로컬 전용(`tools/.review_queue.json`, gitignore).
- ✅ **단건 재임베딩** `tools/reembed_note.py`: 그 노트만 `where={path}` 삭제 후 노트로컬 id로 재적재(전체 리빌드 회피).
  02의 청킹 재사용(빌드와 동일). ⛔재임베딩 전 Chroma 자동 백업(`<db>.bak.<날짜>`)+롤백 경로 출력.
  검증: 사본(chroma.test)에서 여비규정 70청크 재적재 3973→3973, 회수 동일(제9조 d=0.2434). 반영은 `pm2 restart`.
- ✅ **검수 CLI** `tools/review_tool.py`: 큐 순서로 노트 렌더(**별표/별지 강조** `>>별표>>` + 원본파일 대조 안내) →
  ⛔ 사람이 키보드로 `approve` 전체 입력해야만 `검수상태: 검수완료`(+검수일/검토자) 기록 + 재임베딩 호출.
  **자동 승인 없음**(비대화형/빈입력은 건너뜀). 읽기 전용 모드 `--show`/`--list`. 검증: 0건 자동 변경.
- ✅ **UI 검수 배지**: 둘러보기(Explorer)·문서 드로어(DocDrawer)에 이미 존재 — 검수상태 필터 + 배지(미검수 orange/검수완료 green).
- ⏳ 남은 것은 **사람의 실제 검수**(큐 상위부터). 에이전트는 도구·순서까지만.

## P1.3 — 별표/별지 1급 객체화 ✅ (인덱싱), VLM 복원은 보류
- **탐지·분리**: `02_chunk_and_embed.py`가 줄머리 `[별표 N]`·`[별지 제N호]` 헤더를 경계로 별표/별지를
  **독립 청크**(조="별표 N"/"별지 제N호", 별표="Y")로 분리. 본문 인용("별표 1의…")은 경계 아님.
  토글 `CHUNK_BYEOLPYO`(A/B용, 기본 on).
- **refs 메타**: 그 별표/별지를 인용하는 조문 목록을 `refs`에 저장(예: 별표 6 → 제20조). 출처/그래프 연결.
- **재인덱싱**: ⛔백업(`tools/chroma.bak.2026-06-20`) 후 임시 DB 빌드→스왑(무중단 최소화), `pm2 restart`.
  부수효과: 별표가 조문에서 빠져 **과대(2048토큰 초과) 청크 49→33** 감소. 라이브가 stale였던 점도 정정(term 84→168).
- **A/B 측정**(같은 볼트, 35문항, strict):

  | | 별표 카테고리 | Hit@3 | Hit@5 | MRR@3 |
  |---|---|---|---|---|
  | 제N조만(before) | 0/4 | 0.829 | 0.886 | 0.700 |
  | **별표 1급(after)** | **3/4** | **0.914** | **0.971** | **0.738** |

  → 별표 질의 0→3 적중, 전체 +8.5%p(Hit@3/@5), 조문 질의 무회귀. 라이브 답변이 `[여비규정 별표 6]`+`[제20조]` 동시 인용.
  남은 1건(b01 여비 지급 구분표)은 제3조 본문이 "여비 지급 구분표"를 직접 언급해 표를 이김 → 리랭커 타깃.
- **VLM 복원 보류**: 깨진/과대 표(예: 이해충돌방지규정 별표 1 ~14k토큰)는 Qwen2.5-VL 재구성 대상이나
  ~16GB 다운로드 필요 → 승인 시 별도 진행. 현재는 1급 분리·refs까지(원문 비변형, 미검수 유지).

## P1.4 — 리랭커 · 하이브리드 검색 🟡

### 하이브리드(BM25+RRF) — ✅ 측정완료, **이득 없어 기본 off**
- `tools/bm25_index.py`(순수 파이썬 BM25 Okapi + 한글 음절 2-gram + 가중 RRF, 무설치/CPU).
  `rag_core.retrieve(hybrid=...)` opt-in. 기본 `RAG_HYBRID=0`(라이브 영향 없음).
- **before/after(31문항, strict):**

  | | Hit@1 | Hit@3 | Hit@5 | MRR | relaxed@1 |
  |---|---|---|---|---|---|
  | 밀집(dense, KURE-v1) | **0.677** | **0.935** | **1.000** | **0.803** | **0.903** |
  | 하이브리드 RRF(동일가중) | 0.645 | 0.871 | 0.935 | 0.762 | 0.839 |
  | 하이브리드 RRF(밀집 2~4:1) | 0.645 | 0.871 | 0.935 | 0.763 | 0.871 |

  → **하이브리드가 오히려 하락**. 밀집 KURE-v1이 이미 상한(@5 1.000)이고, 의미형 질의에서
  BM25의 '어휘는 비슷하나 틀린' 청크를 RRF가 충분히 누르지 못함(밀집 가중을 올려도 동일).
  ⛔ 가드레일대로 **개선 없으면 켜지 않는다.** BM25/하이브리드는 **opt-in 인프라로 보존**
  (규정명·별표번호·코드처럼 정확 매칭이 중요한 질의군이 평가셋에 늘면 재평가). latency: BM25 인덱스
  1회 구축 ~1–2s(CPU), 질의당 추가 수 ms.

### 리랭커(cross-encoder) — ⛔ 다운로드 승인 대기
- 후보: **BAAI/bge-reranker-v2-m3**(XLM-RoBERTa-large 기반, **약 2.2GB**, HuggingFace, **온프레미스 로컬 추론·외부 API 없음**).
  현재 HF 캐시에 **미존재** → 신규 다운로드 필요. GPU1 여유(현재 24GB 비어있음) 또는 CPU 추론 가능.
- 계획: 밀집 top-N=20 → (q, chunk) 교차인코더 재점수 → top-k=5. `rag_core.retrieve`에 플래그.
  **strict@1 0.677 ↔ relaxed@1 0.903 격차**("규정 맞고 조 틀림")가 리랭커의 핵심 타깃.
- ⛔ 가드레일: 큰 모델 다운로드 전 **무엇/용량/온프레미스 한 줄 보고 후 진행**(이 항목). 승인 시 진행.
