# 12 · 품질 강화 (평가 · 검수 · 별표 · 리랭커) — P1.1~P1.4

> RAG LLM을 "그럴듯함"에서 **측정 가능한 정확함**으로 끌어올리는 품질 하드닝 트랙.
> WORKPLAN의 P0~P5(레포→운영 파이프라인)와 별개로, 운영 중 품질을 끌어올리는 **P1.1~P1.4** 작업이다.
> 코드: [`../eval/`](../eval/README.md) · 기준: [05-rag-design.md](05-rag-design.md) · 거버넌스: [07-security-governance.md](07-security-governance.md).

## ⛔ 이 트랙의 절대 가드레일
1. 규정 값(금액·한도·기한)·정답을 **지어내지 않는다.** 평가셋 정답·기대출처는 **실제 청크에서 추출**, 사람 확정 전 `verified:false`.
2. **검수 완료(`검수상태`) 표시는 사람만 한다.** 에이전트는 큐·도구·후보만 만들고 **자동 확정 금지**.
3. 재임베딩/재인덱싱 전 **Chroma 백업 필수**: `cp -r tools/chroma tools/chroma.bak.$(date +%F)`. 롤백 경로 기록.
4. 큰 모델(리랭커 등) 다운로드 전 **무엇/용량/온프레미스 여부 한 줄 보고** 후 진행. **외부 API 금지**.
5. 별표 등 원문 **비변형**(숫자·표 의역 금지). OCR/VLM 불확실하면 `검수상태:미검수` 유지.
6. **개선 주장은 평가셋 지표(before/after) 없이는 금지.**

## P1.1 — 평가셋 · 평가 하베스트 ✅
- **평가셋** `eval/golden.jsonl`(gitignore, 로컬 전용): 31문항 / 9개 카테고리, 전부 `verified:false`(미검수 초안).
  기대출처(`expected_sources`)는 코퍼스 실재 `(규정명, 조)`에서만 도출 — 규정 값 무추측.
- **하베스트** `eval/run_eval.py` + `eval/run.sh`: 각 질문을 실제 `rag_core.retrieve`에 통과 →
  - Hit@k / Recall@k / MRR (**strict**=규정명+조, **relaxed**=규정명만), 카테고리별 분해, 질문별 상세.
  - 선택 `--judge`: 생성 답변의 **근거 충실도**(근거 밖 사실 단정 여부)·출처표기·면책문구 LLM 판정.
  - 리포트 `eval/reports/<ts>.json`(gitignore). 평가는 **CPU**로 돌려 라이브 서빙(GPU)과 비경쟁.
- **베이스라인**(KURE-v1 + Chroma, 31문항):

  | cutoff | Hit(strict) | Recall | MRR | Hit(relaxed) |
  |---|---|---|---|---|
  | @1 | 0.677 | 0.677 | 0.677 | 0.903 |
  | @3 | 0.935 | 0.935 | 0.790 | 1.000 |
  | @5 | 1.000 | 1.000 | 0.803 | 1.000 |

  → 올바른 **규정**은 거의 top-1(0.903), 정확한 **조**는 top-3(0.935)/top-5(1.000)에 회수.
  **strict@1(0.677) ↔ relaxed@1(0.903)** 격차 = "규정 맞고 조 틀림" → **리랭커(P1.4) 개선 목표 지점**.

- **생성 품질**(LLM-judge, 적중 31 + 부정 5): 근거충실 0.935 · 출처표기 0.968 · **면책문구 0.806** · 거부율 0.800 · 근거없는 조문인용 0건.
- **관찰(사람 확인) → 조치:**
  1. ✅ **[조치완료] 면책 문구 누락 ~19%**(0.806): 14B가 마무리 "최종 판단은…"을 종종 빠뜨림 →
     `rag_core._ensure_disclaimer`로 **answer/answer_stream 모두 결정적 보장(0.806→1.000 by construction)**.
     단위테스트 `tools/test_rag_core.py`(6건) + 라이브 3100 확인. ⛔가드레일 강화(약화 아님).
  2. **환각 1건(n03)**: 길라잡이에 동호회 '명단'만 있고 금액이 없는데 "월 1만원"을 지어냄 → 거부 실패. 가장 위험한 모드를 평가셋이 포착. 근거 충실 강화(프롬프트/리랭커)로 줄일 대상.
  3. **LLM-judge 자체 오판 2건**(q004·q005): 근거에 실제 존재하는 사실을 judge가 '없음'으로 오판 → faithfulness는 **스크리닝 신호**일 뿐 확정은 사람. judge 프롬프트 개선 여지.

> 보안: `golden.jsonl`·`reports/`는 규정 원문 값을 담아 **공개 레포에 커밋하지 않는다**(`chroma`/`app.db`와 동급 로컬 데이터). 형식은 `eval/golden.example.jsonl`.

## P1.3 — 별표/별지 1급 객체화 ✅ (인덱싱), VLM 복원은 보류
- **탐지·분리**: `02_chunk_and_embed.py`가 줄머리 `[별표 N]`·`[별지 제N호]` 헤더를 경계로 별표/별지를
  **독립 청크**(조="별표 N"/"별지 제N호", 별표="Y")로 분리. 본문 인용("별표 1의…")은 경계 아님.
  토글 `CHUNK_BYEOLPYO`(A/B용, 기본 on).
- **refs 메타**: 그 별표/별지를 인용하는 조문 목록을 `refs`에 저장(예: 별표 6 → 제20조). 출처/그래프 연결.
- **재인덱싱**: ⛔백업(`tools/chroma.bak.YYYYMMDD`) 후 임시 DB 빌드→스왑(무중단 최소화), `pm2 restart`.
  부수효과: 별표가 조문에서 빠져 **과대(2048토큰 초과) 청크 49→33** 감소. 라이브가 stale였던 점도 정정(term 84→168).
- **A/B 측정**(같은 볼트, 35문항, strict):

  | | 별표 카테고리 | Hit@3 | Hit@5 | MRR@3 |
  |---|---|---|---|---|
  | 제N조만(before) | 0/4 | 0.829 | 0.886 | 0.700 |
  | **별표 1급(after)** | **3/4** | **0.914** | **0.971** | **0.738** |

  → 별표 질의 0→3 적중, 전체 +8.5%p(Hit@3/@5), 조문 질의 무회귀. 라이브 답변이 `[여비규정 별표 6]`+`[제20조]` 동시 인용.
  남은 1건(b01 여비 지급 구분표)은 제3조 본문이 "여비 지급 구분표"를 직접 언급해 표를 이김 → 리랭커 타깃.
- **VLM 복원 보류**: 깨진/과대 표(예: 이해충돌방지규정 별표 1 ~14k토큰)는 Qwen2.5-VL 재구성 대상이나
  ~16GB 다운로드 필요 → 승인 시 별도 진행. 현재는 1급 분리·refs까지(원문 비변형, 미검수 유지).

## P1.4 — 리랭커 · 하이브리드 검색 🟡

### 하이브리드(BM25+RRF) — ✅ 측정완료, **이득 없어 기본 off**
- `tools/bm25_index.py`(순수 파이썬 BM25 Okapi + 한글 음절 2-gram + 가중 RRF, 무설치/CPU).
  `rag_core.retrieve(hybrid=...)` opt-in. 기본 `RAG_HYBRID=0`(라이브 영향 없음).
- **before/after(31문항, strict):**

  | | Hit@1 | Hit@3 | Hit@5 | MRR | relaxed@1 |
  |---|---|---|---|---|---|
  | 밀집(dense, KURE-v1) | **0.677** | **0.935** | **1.000** | **0.803** | **0.903** |
  | 하이브리드 RRF(동일가중) | 0.645 | 0.871 | 0.935 | 0.762 | 0.839 |
  | 하이브리드 RRF(밀집 2~4:1) | 0.645 | 0.871 | 0.935 | 0.763 | 0.871 |

  → **하이브리드가 오히려 하락**. 밀집 KURE-v1이 이미 상한(@5 1.000)이고, 의미형 질의에서
  BM25의 '어휘는 비슷하나 틀린' 청크를 RRF가 충분히 누르지 못함(밀집 가중을 올려도 동일).
  ⛔ 가드레일대로 **개선 없으면 켜지 않는다.** BM25/하이브리드는 **opt-in 인프라로 보존**
  (규정명·별표번호·코드처럼 정확 매칭이 중요한 질의군이 평가셋에 늘면 재평가). latency: BM25 인덱스
  1회 구축 ~1–2s(CPU), 질의당 추가 수 ms.

### 리랭커(cross-encoder) — ✅ 적용(GPU1)
- **모델**: BAAI/bge-reranker-v2-m3(XLM-RoBERTa-large, ~2.2GB, 온프레미스 로컬추론, 외부 API 없음). 사용자 승인 후 다운로드.
- **파이프라인**: 밀집 top-20 → (질의,청크) 교차인코더 재점수 → top-5. `rag_core.retrieve(rerank=...)` / 환경변수 `RAG_RERANK`.
- **before/after(35문항, strict, 같은 운영 인덱스):**

  | | Hit@1 | Hit@3 | Hit@5 | MRR@3 | 별표 | relaxed@1 |
  |---|---|---|---|---|---|---|
  | 밀집(before) | 0.600 | 0.914 | 0.971 | 0.738 | 3/4 | 0.914 |
  | **+리랭커(after)** | **0.829** | **0.971** | **1.000** | **0.890** | **4/4** | 0.943 |

  → **strict Hit@1 +22.9%p, MRR@3 +15.2%p, 별표 3/4→4/4**. "규정 맞고 조 틀림"을 정조준해 해소.
  P1.3에서 남았던 b01(여비 지급 구분표 vs 제3조 본문)도 리랭커가 표를 1위로 올려 해결. 라이브 답변이 [별표 1] 인용 확인.
- **지연(latency)**: 20쌍 재점수 GPU1 **~0.5s/질의**(CPU는 ~14s라 부적합). LLM 생성 시간 대비 무시할 수준.
- **운영 적용**: `tools/ecosystem.config.js`에 `RAG_RERANK=1`, `RAG_RERANK_DEVICE=cuda:1`(빈 GPU1, 가득 찬 GPU0과 분리).
  기동 `warmup`에서 리랭커 예열. ⛔ **실패(GPU OOM 등) 시 밀집 순서로 우아하게 강등** → API 절대 안 깨짐. 끄려면 `RAG_RERANK=0`.

## P1.5 — 멀티턴 쿼리 재작성(query rewriting) ✅ 적용
후속 질문("몇 퍼센트야? 얼마 줘?")은 단독으로 검색하면 맥락이 빠져 엉뚱한 청크를 회수한다(실제: 초과근무
질의의 후속이 **여비규정 별표2 항공마일리지 50%** 를 회수해 환각). 검색 직전에 직전 대화로 **독립 검색어로 재작성**한다.

- `rag_core.condense_query(question, history)`: history 있으면 LLM으로 후속질문을 독립 질문 한 줄로 재작성 →
  그 검색어로만 검색. ⛔ **답변 생성은 원 질문/근거 그대로**(가드레일·사실성 불변). 첫 턴(history無)·실패 시 원 질문으로 강등.
- 배선: `app_api`(상태형 채팅)·`04_rag_api`(무상태 멀티턴) 둘 다 검색 직전에 호출. 환경변수 `RAG_QUERY_REWRITE`(기본 on).
- **before/after(멀티턴 4문항, 규정명 적중 @3):** before 2/4 → **after 3/4**. 핵심 케이스(mt1 "몇 퍼센트야?")가
  쓰레기 회수(`위탁연구계약/상조회`) → **정상**(`보수규정 제15조·연장근로등관리규칙 제6조`)으로 교정. mt4도 False→True.
  라이브 재현: turn2가 더 이상 항공마일리지로 새지 않고 초과근무 주제 유지, 없는 %는 환각 대신 "확인되지 않습니다".
- **비용/한계**: 후속 턴마다 LLM 1콜 추가(~1–2s). 첫 턴은 비용 0. 재작성이 가끔 특정 키워드를 흘릴 수 있어(mt4
  "파견수당"→"예산 범위") 완벽치 않으나 최악 케이스(맥락 소실)를 분명히 개선. 평가 회귀: `eval/run.sh --rewrite`.

## P2.1 — 답변 피드백 루프(👍/👎) → 검수 우선순위 환류 ✅ 구현·검증
오프라인 평가셋(P1.1)·정적 검수 큐(P1.2)는 "현장에서 실제로 틀린 답"을 모른다. 답변에 **👍/👎(+사유)**를
받아 `app.db`에 저장하고, 자주 👎 받은 규정을 **검수 큐 상단**으로 끌어올린다. 시스템이 실사용으로 스스로 좋아지는 토대.

- **흐름**: UI 👍/👎 → `POST /app/messages/{id}/feedback`(`Feedback` 테이블, 사용자·메시지당 1건·upsert/toggle, 소유격리)
  → `tools/feedback_export.py`(읽기전용 집계) → `tools/.feedback_signals.json` → `tools/review_queue.py --feedback`가 `+min(2·down,20)` 가산.

## P2.3 — 긴 청크 하위청킹(임베딩 잘림 제거) ✅ 구현·검증
임베딩 모델 입력 한도(`max_seq_len=2048`)를 넘는 청크는 **뒷부분이 벡터에 반영되지 않아**(잘림) 그 부분(흔히
항·호의 금액·조건·서약 문구)만 묻는 질의가 회수되지 않았다(기존 28개 청크). 이를 **항(①②…)→호(1./가.)→문단→
줄/문장** 순서로 하위분할해 모든 조각이 한도 이하가 되게 한다(`02_chunk_and_embed.py`의 `subsplit_long_chunks`).

- **불변식 유지**: 하위 조각은 **조 라벨('제N조')·규정명·검수상태 등 메타 그대로**, 하위 인덱스만 `부분`(2/3 등)
  메타로 기록. 조문 조각엔 조 헤더(`제N조(…)`)를 prefix해 자족적. → 출처 `[규정명 제N조]`·드로어 앵커·평가
  strict 매칭(규정명+조) **전부 불변**. 토글 `CHUNK_SUBSPLIT`(기본 on).
- ⛔ **별표/별지(표)는 분할 안 함** — 표 구조 보존. 깨진 거대 표(예: 이해충돌 별표1 ~14k토큰)는 VLM 복원 트랙(P1.3)으로.
- **재색인**: Chroma 백업(`chroma.bak.20260621`) 후 클린 리빌드. 4345 → **4418 청크(+73)**. 재색인 후 한도초과는
  **별표/별지 4개만** 남음(전부 VLM 트랙·의도적 제외) — 조문/머리말/가이드 잘림 0.
- **before/after(44문항 dense, 같은 평가셋)**: Hit@1 0.564 / @5 0.897 → **동일(회귀 0)**. 골든셋이 잘린 꼬리를
  직접 묻지 않아 집계는 평탄 — 효과는 아래 A/B로 입증.
- **A/B 입증(구 백업 vs 신 인덱스, 같은 꼬리 질의)**: `정보보안 기본지침 제1조`(6분할된 긴 조문)의 **2048토큰
  이후 꼬리(보안서약 조항)** 내용으로 검색 → **구 인덱스 10위 밖(미회수) → 신 인덱스 1위**. 잘려서 못 찾던
  내용이 회수됨을 직접 확인. 라이브 재현: "정보보안 서약 위반 시?" → 근거에 `정보보안 기본지침 제1조` 포함.

## P2.2 — 금액·한도 신뢰 강화 + 규정집 기준일 ✅ 구현·검증
행정 사고의 대부분은 **틀린 금액·한도**에서 난다(절대 규칙 1). 생성 LLM의 숫자를 그대로 믿게 하지 말고,
**원문에서 직접 확인**하도록 UI가 유도한다. 재임베딩 없이 프론트(`docdata` 활용)로만 구현.

- **금액·한도 답변 경고**: 답변에 금액/한도 토큰(`원·만원·%·한도·상한·지급액` 등)이 있으면 답변 아래
  "💰 금액·한도가 포함된 답변입니다. 정확한 수치는 우측 근거 원문에서 확인하세요" 안내(클릭 시 근거 패널 활성).
- **근거 스니펫 수치 강조**: 근거 카드 스니펫의 금액/수치(`197,000원`, `1박 2일`, `4시간`…)를 `<mark>`로 강조 →
  사용자 눈이 생성문이 아니라 **원문 수치**로 향한다. 클릭하면 드로어가 해당 조문으로 점프(기존 기능).
- **근거별 검수상태 배지**: 각 근거에 `검수완료`(✓)/`미검수` 배지(`docdata`의 `검수상태` 조회). 전건 미검수인
  현 단계에선 "이 근거는 아직 사람 검수 전"임을 명확히 알려 과신을 막는다(P1.2 검수 진행되면 자동 ✓로 전환).
- **규정집 기준일**: footer 구석에 "📑 규정집 기준일 2026.06.19"(규정 원문 적재일). 규정은 개정되므로 답변이
  **어느 시점 규정**을 근거로 하는지 알린다. 단일 출처 `web/lib/site.ts`의 `CORPUS_AS_OF`(재적재 시 이 값만 갱신).
- **검증**: `web/verify-trust.mjs`(footer 기준일·금액 경고·수치 강조 mark·검수 배지, Playwright 통과).
  headless에 한글(Noto Sans KR·나눔고딕)+이모지(Noto Color Emoji) 폰트 설치 → 스크린샷에 한글·💰·배지까지 충실 표기.

## P2.4 — ERP·서식 연결(규정→실제 처리) ✅ 구현·검증
"규정은 알겠는데 ERP 어디서·무슨 서식으로 처리하지?"가 가장 흔한 후속 질문이다. 답변과 근거 패널이
**처리 위치(ERP 메뉴·서식)**까지 안내하게 한다. (밀집 검색이 이미 ERP/가이드/용어를 함께 회수하므로 — 아래 참고 — 재임베딩 불필요.)

- **답변에 ERP 경로 안내**: 컨텍스트의 시스템(ERP) 청크에 `(ERP 시스템)` 라벨을 붙이고, SYSTEM 프롬프트에
  "[근거]에 '(ERP 시스템)' 항목이 있으면 그 메뉴·경로를 함께 안내(근거에 없는 경로·서식명은 지어내지 않음)" 추가.
  ⛔ 가드레일 불변(무환각). 라이브: "경조사비…ERP 어디서?" → **"회계관리 모듈 → 경비지급의뢰서 작성·기안(화면ID act_2014M)"** 안내.
- **출처 type + UI 칩**: `rag_core._src`에 `type`(regulation|guide|system|term) 추가 → 근거 카드에 **🖥 ERP**(시스템)·
  **📄 서식**(별표/별지) 칩. 클릭 시 드로어로 해당 ERP 노트/서식 점프(기존 기능 재사용).
- **섹션 다양성(opt-in, 기본 off)**: 규정이 top-k를 독점할까 봐 ERP/가이드 좌석 보장 로직(`_select_diverse`,
  `RAG_SECTION_DIVERSITY`)을 만들었으나 **측정 결과 이득 없음**(off=on 동일) — 밀집(KURE-v1)이 이미 규정+가이드+
  시스템+용어를 골고루 회수하기 때문. 하이브리드(P1.4)와 같은 판단으로 **기본 off·인프라만 보존**.
- **검증**: 라이브 답변(ERP 경로·화면ID 포함) + `web/verify-erp-chat.mjs`(답변 ERP 언급 + 근거 🖥 ERP 칩, Playwright 통과).

## P2.5 — 운영자 대시보드(콘텐츠 갭 가시화) ✅ 구현·검증 · 🔒 개인정보 보호 설계
P2.1 피드백·실사용 로그를 **운영자 화면**으로 모아 "다음에 무엇을 보강할지"를 데이터로 보여주되,
**관리자도 개인 채팅은 들여다볼 수 없게** 설계했다. `/admin`(관리자 전용)에 운영 대시보드 섹션 추가.

### 🔒 개인정보 모델 (E2E 암호화 대신 데이터 최소화 + k-익명성)
- **왜 진짜 E2EE가 아닌가**: 서버사이드 RAG는 LLM이 **질문 평문을 읽어야** 답한다 → Signal식 종단간
  암호화는 구조적으로 불가. 그래서 "관리자가 개인 채팅을 못 본다"는 목표를 아래 3가지로 달성한다.
- **개인 채팅 접근 불가**: 관리자에게 타인 채팅을 읽는 엔드포인트가 없다(`get_chat`는 소유자 검증). 본인만 자기 기록을 본다.
- **본문 비노출**: `GET /app/stats`·`GET /app/feedback`는 질문·답변 **원문을 절대 반환하지 않는다**(규정 메타·집계만).
- **k-익명성**: 인기 질문·콘텐츠 갭은 서로 다른 사용자 **K명 이상**(`STATS_MIN_USERS`, 기본 3)이 물은 항목만,
  그것도 **사용자 수 집계**로만 노출. 1~2명의 고유·민감 질문은 숨겨지고 누가 물었는지도 알 수 없다.

### 기능
- **집계 API** `GET /app/stats?days=`(관리자 전용): 활동 요약(사용자·대화·질문·답변), **거부율**(`REFUSAL_RE` 감지),
  **피드백**(👍/👎), **인기 질문**·**콘텐츠 갭**(k-익명 — 거부된 답변의 직전 질문 중 K명 이상). `GET /app/feedback`는 규정·사유만.
- **UI**: 요약 카드 + 인기 질문/콘텐츠 갭 2단 + 🔒 개인정보 안내 배너. 정적 export 런타임 fetch, 비관리자 403→안내.
- **루프 완성**: 인기/거부/👎 신호(집계) → 검수 큐(P2.1 `feedback_export`) + 콘텐츠 로드맵.
- **검증**: 백엔드 `tools/test_stats.py`(k-익명: 2명 질문 노출·1명 질문 숨김·응답에 개인 원문 0·관리자 403),
  `tools/test_feedback.py`(피드백 응답에 질문/답변 본문 미노출 확인), 프론트 `web/verify-dashboard.mjs`(개인정보 배너·카드·플래그 공존, Playwright).

## P2.6 — 두괄식 답변(결론 먼저) ✅ 구현·검증
사용자 피드백 "결론이 먼저 나오고 근거를 알려주면 좋겠다"에 대응. 기존 SYSTEM은 "단계로 설명"만 지시해
답변이 나열·절차부터 시작하고 결론이 묻혔다(실측: 5/5 미달). `rag_core.SYSTEM` 규칙을 두괄식으로 교체.

- **규칙2(신규)**: 답변 맨 앞에 핵심 결론을 한두 문장 **굵게(`**…**`)** 먼저. 금액·가부·기한이면 그 값/가부를 첫 문장에 직접. 절차 나열은 결론 뒤.
- **규칙3(보강)**: 결론은 짧게, **본문은 충실하게** — 구체적 처리 방법(신청 메뉴·화면 경로·단계·필요 서식·금액)을 빠뜨리지 말 것(두괄식이 본문을 과하게 압축하지 않도록).
- **규칙4**: 출처는 핵심 조문을 맨 앞에(피드백 #1 "어느 근거 볼지" 완화). 거부·면책·ERP 무환각 규칙은 그대로.
- **before/after(5문항, git stash로 구/신 프롬프트 대조)**: before 0/5 → **after 5/5** 굵은 결론 선두.
  거부 케이스도 "**규정에서 확인되지 않습니다.**" 두괄식. 1차 변경 시 경조사 답변에서 ERP 경로가 누락돼(과압축)
  규칙3을 보강하니 결론+ERP 경로("회계(ACT)>경비지급의뢰서")·별지 5호 서식·금액까지 본문에 복원.
- **검증**: 거부 가드레일 유지(근거 밖 질문 2건 모두 거부), `web/verify-dugwal.mjs`(채팅 UI 첫 줄이 굵은 결론으로 시작, Playwright 통과).
- **무영향**: 검색(retrieve)은 손대지 않아 Hit/Recall/MRR 불변 — 생성 형식만 개선. 재임베딩 불필요.
